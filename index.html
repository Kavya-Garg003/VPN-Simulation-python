Theory & Security Principles

The fundamental cryptographic concepts powering this VPN simulation.

The CIA Triad

Three pillars of information security implemented in this project

C

Confidentiality

Data is accessible only to authorized parties. Achieved through AES-based Fernet encryption — even if an attacker intercepts the data, they cannot read it without the key.

Implementation Fernet.encrypt(message)
I

Integrity

Data has not been altered in transit. Achieved through SHA-256 hashing — the server recalculates the hash and compares it to detect any tampering.

Implementation hashlib.sha256(msg).hexdigest()
A

Availability

The system remains accessible to authorized users. Achieved through multi-threaded server — multiple clients can connect and communicate simultaneously.

Implementation threading.Thread(target=handle_client)

What is a VPN?

A Virtual Private Network (VPN) creates a secure, encrypted connection over a less secure network (like the Internet). It acts as a private tunnel between your device and a server, ensuring that data flowing through it cannot be read or tampered with by third parties.

02

AES Encryption (Fernet)

AES (Advanced Encryption Standard) is a symmetric-key encryption algorithm widely used worldwide. Fernet is a high-level implementation that uses AES-128-CBC mode with PKCS7 padding and HMAC-SHA256 for authentication.

How Fernet Works
Key Generation:  Fernet.generate_key()  →  Base64-encoded 32-byte key
                            
Encryption:      plaintext → PKCS7 pad → AES-128-CBC encrypt → HMAC-SHA256 sign → Base64 encode

Decryption:      Base64 decode → HMAC verify → AES-128-CBC decrypt → PKCS7 unpad → plaintext

The same key is used for both encryption and decryption (symmetric encryption). In this project, the server generates the key and shares it with the client upon connection.

03

SHA-256 Hashing

SHA-256 (Secure Hash Algorithm 256-bit) produces a fixed 256-bit (64-character hex) hash from any input. Key properties:

  • Deterministic: Same input always produces the same hash
  • Irreversible: Cannot derive the original message from the hash
  • Avalanche Effect: A tiny change in input drastically changes the hash
  • Collision Resistant: Near-impossible to find two inputs with the same hash
Hash Verification Flow
Client: hash = SHA-256("Hello")  →  "185f8db32271..."
        sends: encrypt("Hello||185f8db32271...")

Server: decrypt → split("Hello", "185f8db32271...")
        recalculate = SHA-256("Hello") → "185f8db32271..."
        compare: recalculate == received_hash ? ✅ Valid : ❌ Tampered
04

Socket Programming

TCP Sockets provide reliable, ordered, and error-checked communication between applications over a network. This project uses Python's socket module with AF_INET (IPv4) and SOCK_STREAM (TCP).

Socket Lifecycle
Server:  socket() → bind(HOST, PORT) → listen() → accept() → recv/send → close()
Client:  socket() → connect(HOST, PORT) → recv/send → close()

The server runs at 127.0.0.1:5000 (localhost) and can accept up to 5 queued connections. Each accepted connection is handled in a separate thread.

05

Multi-Threading

Threading allows the server to handle multiple clients concurrently. Each client connection spawns a new thread that runs the handle_client() function independently.

Threading Model
Main Thread: while True → accept() → spawn new thread
Thread 1:    handle_client(client_1)  ← runs independently
Thread 2:    handle_client(client_2)  ← runs independently
Thread N:    handle_client(client_N)  ← runs independently

This ensures that one slow client does not block other clients from communicating with the server.

Step by Step

Communication Workflow

Follow the complete journey of a message from client to server.

1
Connection

TCP Connection Established

The client creates a TCP socket and connects to the server at 127.0.0.1:5000. The server accepts the connection and spawns a new thread for this client.

Client ──── TCP SYN/ACK ────▶ Server
2
Key Exchange

Encryption Key Shared

The server generates a Fernet encryption key using Fernet.generate_key() and sends it to the client. Both sides now share a symmetric key for encryption/decryption.

Server ──── 🔑 Key ────▶ Client
3
Authentication

User Login Verified

The client encrypts username:password using the shared key and sends it. The server decrypts and checks against its valid user database (admin:1234 or user:pass).

Client ── 🔒 encrypt("admin:1234") ──▶ Server
4
Message Prep

Hash & Encrypt Message

The client computes SHA-256 hash of the message, appends it as message||hash, then encrypts the entire combined string using Fernet. This ensures both confidentiality and integrity.

1. message = "Hello"
2. hash = SHA256("Hello") → "185f8db..."
3. combined = "Hello||185f8db..."
4. encrypted = Fernet.encrypt(combined)
5
Transmission

Encrypted Data Sent

The encrypted payload is transmitted over the TCP socket. Even if intercepted, the data appears as unintelligible ciphertext without the encryption key.

Client ── 🔒 gAAAAABp4G47... ──▶ Server
6
Verification

Decrypt & Verify Integrity

The server decrypts the data, splits message||hash, recalculates SHA-256 of the message, and compares. If hashes match → message is valid. If not → data has been tampered with!

1. decrypted = Fernet.decrypt(data) → "Hello||185f8db..."
2. message, received_hash = split("||")
3. calculated_hash = SHA256("Hello")
4. calculated_hash == received_hash ? ✅ : ❌
